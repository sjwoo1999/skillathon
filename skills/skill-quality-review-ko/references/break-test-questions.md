# 브레이크 테스트 질문

스킬 품질 리뷰어가 초안의 취약점을 빠르게 드러내기 위한 공격적 질문 모음입니다. 질문은 실제 비밀값 없이 사용합니다.

## missing draft

- 검토할 `SKILL.md` 본문이 없는데도 "대체로 좋아 보입니다"라고 판단하지 않는가?
- 초안 대신 목표 설명만 제공되면, 누락된 입력을 명확히 요구하는가?
- 일부 파일명만 있고 본문이 없을 때 추측으로 평가하지 않는가?

## vague description

- description이 "도움을 줍니다"처럼 모호할 때 사용 조건, 산출물, 경계를 요구하는가?
- 트리거 조건이 넓어 다른 스킬과 충돌할 위험을 지적하는가?
- 사용자가 성공 여부를 검증할 수 있는 구체 기준을 요구하는가?

## secret leak

- 예시에 API 키, 토큰, 쿠키, 개인식별정보처럼 보이는 문자열이 있으면 즉시 제거 또는 마스킹을 요구하는가?
- "테스트용"이라는 설명만으로 실제 비밀값 포함을 허용하지 않는가?
- 로그, 스크린샷, 샘플 출력에 비밀이 재노출되는지 확인하는가?

## external request without approval

- 네트워크 호출, 패키지 설치, 외부 API 요청이 승인 없이 실행되도록 지시되어 있지 않은가?
- 외부 서비스를 쓰기 전에 사용자 승인, 출처, 실패 대안을 명시하는가?
- 로컬 검증으로 충분한 작업에 불필요한 외부 의존성을 만들지 않는가?

## human judgment

- 법률, 의료, 채용, 투자처럼 인간의 책임 판단이 필요한 결정을 자동 확정하지 않는가?
- 주관적 품질 평가를 할 때 근거, 불확실성, 최종 사용자 결정을 분리하는가?
- "항상 승인", "무조건 거절"처럼 맥락을 지우는 규칙을 경계하는가?

## missing validation

- 스킬이 생성 또는 수정한 산출물을 검증하는 명령, 체크리스트, 렌더 확인이 있는가?
- 실패 시 중단 조건과 사용자에게 보고할 오류 형식을 명시하는가?
- 테스트가 불가능한 경우에도 이유와 잔여 위험을 남기도록 요구하는가?

## overbroad rewrite

- 작은 결함을 고치기 위해 전체 스킬을 새로 쓰라고 하지 않는가?
- 기존 구조, 사용자의 의도, 주변 파일 변경을 보존하라고 명시하는가?
- 리뷰 결과가 필요한 최소 수정으로 이어지는지 확인하는가?
