# 출력 템플릿

최종 답변은 한국어로 작성하고, 발견 사항을 먼저 배치한다. 근거가 약한 평가는 낮은 심각도로 두거나 질문으로 남긴다.

## 총평

- 전체 판정: `강함`, `보통`, `위험`, `차단`
- 한 줄 요약: 가장 중요한 품질 리스크와 다음 행동

| 입력 계약 | 강함/보통/위험 | 문서상 증거 | 예상 실행 영향 |

## 발견 사항

### Blocker

- 릴리스나 사용을 막는 문제만 적는다.
- 각 항목은 `문제`, `증거`, `영향`, `권장 수정`을 포함한다.

### Warning

- 실패 가능성이 크거나 반복 사용에서 혼란을 만들 문제를 적는다.
- 각 항목은 `문제`, `증거`, `영향`, `권장 수정`을 포함한다.

### Improvement

- 품질을 높이지만 차단 수준은 아닌 개선점을 적는다.
- 작고 구체적인 편집 제안으로 작성한다.

## 우선순위 작은 수정

1. 가장 적은 변경으로 Blocker를 없애는 수정
2. Warning을 줄이는 문장 또는 섹션 보강
3. 재사용성과 검증 가능성을 높이는 정리

## 깨뜨려 볼 질문

- 필수 입력이 빠졌을 때 스킬이 멈추고 요청하는가?
- 서로 충돌하는 지시가 있을 때 우선순위가 드러나는가?
- 제공 자료 밖의 사실을 단정하지 않도록 막는가?
- 최종 답변 형식이 매번 같은 구조로 재현되는가?

## 재검증 방법

- 수정 후 동일 입력으로 다시 평가하고 Blocker가 사라졌는지 확인한다.
- 최소 1개의 정상 입력과 1개의 결핍 입력으로 드라이런한다.
- 출력이 루브릭, 발견 사항, 수정안, 재검증을 모두 포함하는지 확인한다.

## 사람이 마지막으로 볼 것

- 평가 범위가 실제 릴리스 목적과 맞는지
- 심각도 판정이 팀 기준과 맞는지
- 권장 수정이 스킬의 의도와 톤을 해치지 않는지
- 법무, 보안, 브랜드처럼 자동 평가로 확정할 수 없는 요구가 남아 있는지
