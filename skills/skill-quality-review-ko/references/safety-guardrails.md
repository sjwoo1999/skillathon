# Skill 품질 평가 안전 가드레일

## Secret/PII 처리

- API 키, 토큰, 비밀번호, 세션 쿠키, 개인식별정보, 내부 URL, 고객 데이터는 민감정보로 간주한다.
- 민감정보가 보이면 값 자체를 기록하지 말고 유형과 위치만 요약한다.
- 평가 예시는 실제 값 대신 `<SECRET>`, `<EMAIL>`, `<CUSTOMER_ID>` 같은 대체 표기를 사용한다.

## 외부 요청 승인 확인

- 네트워크 요청, 파일 업로드, 외부 API 호출, 원격 저장소 접근이 필요한 경우 사전 승인 조건이 있는지 확인한다.
- 승인 지침이 없으면 외부로 데이터를 보내지 않는 것을 기본값으로 둔다.
- 승인 여부를 평가할 때는 목적, 전송 대상, 전송 데이터 범위, 실패 시 영향이 명시되었는지 본다.

## 인간 판단 영역

- 법률, 의료, 금융, 채용, 학업 평가, 보안 사고 대응처럼 중대한 판단이 필요한 영역은 자동 결론을 제한해야 한다.
- Skill은 근거 정리, 위험 표시, 검토 체크리스트 제공까지를 기본 역할로 삼고 최종 판단자는 사람임을 명시해야 한다.
- 확정적 조언, 보장, 전문 자격을 암시하는 표현은 위험 신호로 본다.

## 중단/제한 조건

- 비밀값 노출을 요구하거나, 무단 접근, 권한 우회, 파괴적 변경, 대량 전송을 요구하면 중단 조건이 있어야 한다.
- 입력이 부족해 안전하게 평가할 수 없으면 추측하지 말고 필요한 정보를 요청해야 한다.
- 위험 작업은 최소 범위, 되돌릴 수 있는 방식, 명시적 승인 여부를 확인한 뒤 진행하도록 제한해야 한다.

## 민감값 재노출 금지

- 사용자 입력이나 파일에서 발견한 민감값을 답변, 로그, 리뷰 코멘트, 예시 코드에 그대로 반복하지 않는다.
- 필요한 경우 마지막 2~4자만 표시하거나 전체를 마스킹한다.
- 민감값 제거를 제안할 때도 원문을 인용하지 말고 저장 위치와 조치만 설명한다.
